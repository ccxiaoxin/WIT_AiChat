# Interactive — 选择题驱动的交互式对话

## 概述

将整个对话过程转变为**持续的选择题交互循环**。Agent 不再单方面输出长篇回答，而是通过一轮又一轮的 AskQuestion 选择题与用户协作推进任务。

**核心理念**：一次 request 内完成多轮交互，每轮都是选择题，用户只需点选即可推进。

## 为什么使用 Interactive 模式？

> **节省 Cursor Request 数量**：传统对话模式下，每次用户输入文字回复都会消耗一次 Request。而 Interactive 模式利用 AskQuestion 工具在**单次 Request 内完成多轮交互**——用户通过点选选项推进对话，不触发新的 Request 计费。对于需要多轮澄清、确认的复杂任务，可显著减少 Request 消耗量。

## 激活条件

- **不自动激活**：仅当用户在指令中明确提到 `interactive` 或 `交互模式` 时才使用此 skill
- 可与其他 skill 组合使用，例如 `interactive + brainstorming`

## 核心规则

### 规则 1：一切皆选择题

- 所有需要用户决策的地方，都转化为选择题
- 每个选择题提供 2-5 个选项，不超过 5 个
- **明确标注单选/多选**：在 prompt 文本末尾标注 `（单选）` 或 `（可多选）`，同时对多选问题设置 `allow_multiple: true`

### 规则 2：分批提问，逐个击破

- 如果有多个问题需要确认，**不要一次性全部抛出**
- 每次 AskQuestion 调用最多包含 **1-3 个问题**
- 按优先级或逻辑依赖关系排序，先问关键问题，后问细节问题
- 前一批问题的答案可能影响后续问题的选项

### 规则 3：永续循环——不停直到用户说停

- 用户回答完当前批次的选择题后，Agent 应：
  1. 简短总结当前进展（1-2 句话）
  2. 立即给出下一批选择题
- **任务完成后不自动停止**。当一个任务/阶段完成后，Agent 必须主动询问：
  - "还有其他需要处理的吗？"
  - 提供续航选项（如：继续优化、处理新任务、回顾已完成内容等）
- **循环仅在以下条件满足时终止**：
  - 用户**明确选择**了"结束交互"或"停止"选项
- **禁止**在用户未主动选择停止时自行结束交互

### 规则 4：每轮必带控制选项

每一轮 AskQuestion 的**最后一个问题**（或唯一问题的最后几个选项）必须包含以下控制选项中的至少一个：

| 控制选项 | 含义 |
|---------|------|
| 跳过此问题 | 暂时跳过，后续可能再问 |
| 全部跳过，开始执行 | 结束讨论，用当前已确认的信息开始执行 |
| 返回上一步 | 修改上一轮的选择 |
| 结束交互 | 停止整个交互流程 |

根据当前阶段选择合适的控制选项，不需要每次都包含全部四个。

### 规则 5：上下文累积与回顾

- Agent 内部维护一个"已确认决策列表"
- 每 3-5 轮交互后，用简短的表格或列表回顾已确认的决策
- 回顾后继续下一轮选择题

## 交互流程模板

### 阶段 1：理解意图

首先确认用户想做什么：

```
AskQuestion({
  title: "开始交互",
  questions: [{
    id: "intent",
    prompt: "你想做什么？（单选）",
    options: [
      { id: "a", label: "<根据上下文推断的选项 A>" },
      { id: "b", label: "<根据上下文推断的选项 B>" },
      { id: "c", label: "<根据上下文推断的选项 C>" }
    ]
  }]
})
```

### 阶段 2：细化需求

根据意图，逐步细化：

- 每轮 1-3 个问题
- 每个问题 2-5 个选项
- 选项应具体、可操作，避免模糊表述

### 阶段 3：确认与执行

所有关键决策确认后：

```
AskQuestion({
  title: "确认方案",
  questions: [{
    id: "confirm",
    prompt: "以上方案是否确认？（单选）",
    options: [
      { id: "execute", label: "确认，开始执行" },
      { id: "adjust", label: "需要调整（返回修改）" },
      { id: "cancel", label: "取消，结束交互" }
    ]
  }]
})
```

### 阶段 4：执行中反馈

执行过程中遇到需要决策的地方，继续用选择题：

```
AskQuestion({
  title: "执行中决策",
  questions: [{
    id: "decision",
    prompt: "<遇到的具体问题>（单选）",
    options: [
      { id: "opt_a", label: "<方案 A>" },
      { id: "opt_b", label: "<方案 B>" },
      { id: "skip", label: "跳过，使用默认" }
    ]
  }]
})
```

### 阶段 5：任务完成续航

当一个任务完成后，**不要结束交互**，主动询问：

```
AskQuestion({
  title: "任务已完成",
  questions: [{
    id: "next_action",
    prompt: "✅ <已完成任务的简述>。还需要做什么？（单选）",
    options: [
      { id: "continue", label: "继续优化/完善刚才的工作" },
      { id: "new_task", label: "处理新的任务" },
      { id: "review", label: "回顾所有已完成的工作" },
      { id: "stop", label: "结束交互，到此为止" }
    ]
  }]
})
```

## 选项设计原则

1. **具体而非抽象**：用"添加 Redis 缓存层"而非"优化性能"
2. **互斥且完整**：选项之间不重叠，覆盖主要可能性
3. **有默认推荐**：在选项 label 中用 `（推荐）` 标注 Agent 建议的选项
4. **渐进式细化**：先粗粒度选择，再细粒度选择
5. **明确单选/多选**：在 prompt 末尾标注 `（单选）` 或 `（可多选）`。当问题语义上允许多个答案时（如 "你关注哪些方面？"），设置 `allow_multiple: true`

## 与其他 Skill 组合

当与其他 skill 组合使用时，interactive 模式覆盖该 skill 的交互方式：

- 原 skill 中需要用户输入文字的地方 → 转为选择题
- 原 skill 中的确认步骤 → 保持选择题
- 原 skill 的核心逻辑和执行步骤 → 不变

## 示例场景

用户说："interactive 帮我重构这个函数"

```
轮次 1: 你想如何重构？→ [拆分子函数 / 简化逻辑 / 提取接口 / 改善命名 / 全部都要]
轮次 2: 优先处理哪个？→ [拆分子函数(推荐) / 简化逻辑 / 让我看看代码再决定]
轮次 3: 拆分策略？→ [按职责拆分 / 按调用频率拆分 / 你来决定(推荐)]
轮次 4: 确认方案 → [确认执行 / 调整 / 取消]
...执行中...
轮次 5: 发现这个变量命名不佳，要一起改吗？→ [改 / 不改 / 稍后再说]
```
